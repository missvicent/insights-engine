## 1. Why this exists

The previous auth design ([2026-04-16](./2026-04-16-supabase-jwt-auth-design.md)) assumed **native Supabase Auth**: HS256 JWTs signed with `SUPABASE_JWT_SECRET`, verified locally, `sub` = Supabase user UUID. That spec explicitly scoped RS256/JWKS out.

Reality is different:

- The app is paired with **Clerk**, not native Supabase Auth.
- Clerk currently signs via its legacy "Supabase JWT Template" — HS256 with the shared Supabase JWT secret. A decoded sample token from 2026-04-23:
  ```json
  // header
  {"alg":"HS256","cat":"cl_B7d4PD222AAA","typ":"JWT"}
  // payload
  {
    "iss": "https://worthy-hornet-72.clerk.accounts.dev",
    "aud": "authenticated",
    "role": "authenticated",
    "sub": "user_33IZQVpXj8DAGKJyec47RxZanBE",
    "exp": 1776970985,
    "iat": 1776970925
  }
  ```
- This legacy path was **deprecated by Clerk in April 2025** in favour of **Clerk as a Third-Party Auth Provider** on Supabase: RS256 with Clerk's own key pair, public keys published via a JWKS endpoint, no shared secret in flight.
- Today's backend decodes the legacy HS256 path only by coincidence — `SUPABASE_JWT_SECRET` happens to match Clerk's custom signing key. There is no `iss` validation, no acknowledgement of Clerk anywhere in code, config, Swagger, or docs.

This migration gets us onto Clerk's modern path and makes Clerk first-class in config and docs.

## 2. Architecture (target state)

```
Client (Clerk SDK)
  │  Authorization: Bearer <clerk-rs256-jwt>
  ▼
FastAPI
  │
  └─ HTTPBearer(scheme_name="ClerkBearer") ← Swagger Authorize dialog
        │
        ▼
  get_user_ctx(credentials)
        │
        │  1. PyJWKClient resolves signing key by `kid`
        │     (HTTPS to {CLERK_ISSUER}/.well-known/jwks.json, cached)
        │  2. jwt.decode(token, key, RS256, iss=CLERK_ISSUER, aud="authenticated")
        │  3. 401 on any failure (expired / issuer / network / bad-sig)
        │
        ▼
  UserContext(user_id=payload["sub"], db=build_user_client(token))
        │
        ▼
  fetch_*(ctx, ...) → PostgREST with Clerk JWT attached
        │
        ▼
  Supabase
    - Third-Party Auth Provider (Clerk) verifies via JWKS link
    - RLS policies compare user_id = auth.uid() (both text)
```

### Trust boundaries

| Layer | Trust anchor | Algorithm | Validated claim(s) |
|---|---|---|---|
| Backend decode | Clerk JWKS URL | RS256 | `iss`, `aud`, `exp`, `sub` |
| Supabase RLS | Clerk JWKS URL (via Third-Party Auth config) | RS256 | `sub` → `auth.uid()` |

Two independent verifications share the same trust anchor (Clerk's public keys) but run in separate processes. Compromising one does not bypass the other — the belt-and-suspenders property the 2026-04-16 spec aimed for is preserved.

### What changes vs. 2026-04-16

| Concern | Old (2026-04-16) | New (this spec) |
|---|---|---|
| Algorithm | HS256 | RS256 |
| Signing key | `SUPABASE_JWT_SECRET` (shared) | Clerk's private key (asymmetric) |
| Backend key lookup | Direct from settings | Cached `PyJWKClient` → HTTPS to Clerk |
| `iss` check | Not performed | **Required**, matches `CLERK_ISSUER` |
| Prod use of `SUPABASE_JWT_SECRET` | Primary | Unused in prod decode; retained only for legacy tests |
| Clerk acknowledgement in code/docs | None | Settings field, OpenAPI description, README section |
| Supabase-side trust | `SUPABASE_JWT_SECRET` matches Clerk's custom key (coincidence) | Supabase Third-Party Auth provider points at Clerk JWKS (explicit) |

### Boundaries preserved

The architecture rules from `CLAUDE.md` still hold: `insights_engine.py` is pure logic, `ai_service.py` receives `InsightSummary` only, routes are thin. Auth is confined to `app/routes/deps.py` + the new `app/auth/jwks.py`. Services and models are untouched by this migration.

<details>
<summary>Reusable audit queries</summary>

**1. Dump every policy body in one shot:**

```sql
SELECT schemaname, tablename, policyname, cmd, qual, with_check
FROM pg_policies
WHERE schemaname = 'public'
  AND tablename IN (
    'budgets', 'transactions', 'goals',
    'debts', 'allocations', 'recurring_transactions'
  )
ORDER BY tablename, policyname;
```

Expected idiom for each `qual` / `with_check`:

```sql
USING (user_id = (auth.jwt() ->> 'sub'))
```

Flag anything that casts to `uuid`, relies on `auth.uid()`, or pulls from a different claim.

**2. End-to-end impersonation** — run as service-role, highlight the entire block, Run once. `set local` only holds inside the transaction, and the Supabase SQL editor opens a fresh session per Run, so splitting this across multiple Runs will give you a NULL `auth.jwt()`:

```sql
begin;
set local role authenticated;
-- auth.jwt() reads the full claims blob from request.jwt.claim / request.jwt.claims,
-- NOT the per-claim GUC request.jwt.claim.sub. Set the whole JSON object.
set local request.jwt.claims =
  '{"sub":"user_33IZQVpXj8DAGKJyec47RxZanBE","role":"authenticated"}';

select
  auth.jwt()                                                     as claims,
  auth.jwt() ->> 'sub'                                           as claim_sub,
  (select user_id from budgets
     where user_id = 'user_33IZQVpXj8DAGKJyec47RxZanBE' limit 1) as row_user_id,
  (select count(*) from budgets)                                 as visible_budgets;

rollback;
```

Interpret the one-row result:

- `claims` NULL ⇒ the `set local` didn't reach this transaction; re-run the block as a single batch.
- `claim_sub` set but `row_user_id` NULL ⇒ the user genuinely has no rows; pick a user that does.
- `claim_sub ≠ row_user_id` ⇒ whitespace/casing drift in one of them.
- All four populated and `visible_budgets > 0` ⇒ policies are doing their job.

### Phase B — backend code (the learning bits)

This phase is yours to implement by hand. Each step describes the target shape and names the key moving parts; use it as a guide, not a copy-paste source.

#### B.1 `[USER]` Extend `Settings` with `clerk_issuer`

**File:** `app/db/client.py`

**Context:** The prod decode path needs to know which issuer to trust, and we want misconfiguration to fail at app boot, not at first request. `pydantic-settings` binds env vars to fields, so declaring `clerk_issuer: str` (no default) makes `AI_MODEL`-style fail-loud behaviour automatic.

**Target shape:**

```python
class Settings(BaseSettings):
    supabase_url: str
    supabase_anon_key: str
    ai_model: str
    clerk_issuer: str                       # required; matches CLERK_ISSUER env
    clerk_jwks_url: str | None = None       # optional; derived from clerk_issuer if unset

    # Legacy — still used by scripts/dev_token.py if you keep it, and by
    # some tests during the transition. Not read in the production decode path.
    supabase_jwt_secret: str | None = None

    class Config:
        env_file = ".env"
        extra = "ignore"
```

**Why optional vs required:**
- `clerk_issuer` → **required**. Without it the decode path has nothing to match `iss` against, so the app should refuse to start.
- `clerk_jwks_url` → **optional**. Most of the time it's `{clerk_issuer}/.well-known/jwks.json`; the factory in B.2 derives it. Overridable in case Clerk ever changes the path or you're pointing at a test server.

**Done when:** `Settings()` raises `ValidationError` if `CLERK_ISSUER` is missing from the environment; otherwise instantiates cleanly.

#### B.2 `[USER]` Create `app/auth/jwks.py`

**Context:** `PyJWKClient` manages the HTTPS fetch to `/.well-known/jwks.json`, caches keys by `kid`, and reuses the same `requests.Session` across calls. You want exactly **one** instance per process so caching works. An `lru_cache(maxsize=1)` factory is the idiomatic Python way.

**Target shape (new file):**

```python
from functools import lru_cache

from jwt import PyJWKClient

from app.db.client import get_settings


@lru_cache(maxsize=1)
def get_jwks_client() -> PyJWKClient:
    s = get_settings()
    url = s.clerk_jwks_url or f"{s.clerk_issuer.rstrip('/')}/.well-known/jwks.json"
    # cache_keys=True keeps keys warm across requests.
    # lifespan (seconds) controls how often PyJWKClient re-fetches the JWKS
    # document; Clerk rotates rarely and overlaps new/old keys during rotation,
    # so an hour is comfortably safe.
    return PyJWKClient(url, cache_keys=True, max_cached_keys=16, lifespan=3600)
```

Also create `app/auth/__init__.py` (empty) so Python treats `app/auth/` as a package.

**Key concepts to internalise while writing this:**
- **Why a factory, not a module-level instance?** `PyJWKClient` does the first fetch lazily on first `get_signing_key_from_jwt` call, not at construction — so building it at import time is fine. But wrapping it in a factory means tests can monkeypatch `get_jwks_client` to return a stub without touching the real HTTP layer.
- **Why `lru_cache` with `maxsize=1` instead of a module-level global?** Same behaviour, but with a convention tests already use (`get_settings()` does the same). Consistency makes the codebase easier to hold in your head.

**Done when:** ` venv/bin/python -c "from app.auth.jwks import get_jwks_client; print(get_jwks_client())"` prints a `PyJWKClient` instance (no network call yet — first fetch is lazy).

#### B.3 `[USER]` Rewrite `get_user_ctx` for RS256

**File:** `app/routes/deps.py`

**Context:** The HS256 code path is replaced. `HTTPBearer` continues to handle the ingest-side `Authorization: Bearer …` parsing (we keep the change from earlier today). The decode path now fetches a signing key from the JWKS cache, then lets `jwt.decode` verify signature + standard claims + issuer in one call.

**Target shape:**

```python
from typing import Annotated

import jwt
from fastapi import Depends, HTTPException
from fastapi.security import HTTPAuthorizationCredentials, HTTPBearer

from app.auth.jwks import get_jwks_client
from app.context import UserContext
from app.db.client import build_user_client, get_settings


bearer_scheme = HTTPBearer(
    auto_error=True,
    scheme_name="ClerkBearer",
    description=(
        "Clerk-issued JWT (Third-Party Auth, RS256, verified against Clerk's JWKS). "
        "Paste the token only — Swagger prepends 'Bearer '. "
        "Clerk tokens typically live ~60s; refresh from the frontend if it expires."
    ),
)


def get_user_ctx(
    credentials: Annotated[HTTPAuthorizationCredentials, Depends(bearer_scheme)],
) -> UserContext:
    settings = get_settings()
    token = credentials.credentials

    try:
        signing_key = get_jwks_client().get_signing_key_from_jwt(token).key
        payload = jwt.decode(
            token,
            signing_key,
            algorithms=["RS256"],
            audience="authenticated",
            issuer=settings.clerk_issuer,
            leeway=5,  # matches Clerk's dashboard "Allowed clock skew: 5s"
            options={"require": ["exp", "sub", "aud", "iss"]},
        )
    except jwt.ExpiredSignatureError:
        raise HTTPException(status_code=401, detail="token expired") from None
    except jwt.InvalidIssuerError:
        raise HTTPException(status_code=401, detail="invalid token issuer") from None
    except jwt.InvalidTokenError:
        raise HTTPException(status_code=401, detail="invalid token") from None
    except Exception:
        # JWKS fetch errors, key-id not found, unexpected cryptography errors:
        # do not leak them as 500. 401 with a generic message is the contract.
        raise HTTPException(status_code=401, detail="unable to verify token") from None

    user_id = payload["sub"]
    if not user_id:
        raise HTTPException(status_code=401, detail="invalid token")
    return UserContext(user_id=user_id, db=build_user_client(token))
```

**Key concepts:**
- **`algorithms=["RS256"]` — not a list including `HS256`.** This is the fix for the classic "algorithm downgrade" attack: an attacker who swaps the token header to `alg: HS256` and signs with the Supabase JWT secret would pass verification if we allowed HS256. By restricting to RS256, PyJWT rejects any non-RS256 token outright.
- **`options={"require": [...]}`** — PyJWT does not require *any* standard claim by default. The `require` list enforces their presence; `audience=` and `issuer=` enforce their value.
- **The final `except Exception`** — specifically catches JWKS network failures and `PyJWKClientError`. Without it, a transient DNS issue surfaces as 500 to the client; with it, we stay in the 401 contract.

**Done when:**
- `_extract_bearer` helper is gone (it was deprecated by the earlier `HTTPBearer` change).
- No import of `Header` remains.
- `app/routes/deps.py` imports `get_jwks_client` from the new `app/auth/jwks.py`.
- Uvicorn boots with the new `Settings` values and `CLERK_ISSUER` set.

## 6. Rollback

If post-flip production breaks and you need to restore service within minutes:

1. **Clerk dashboard** → JWT Templates → `supabase` → turn **Custom signing key** back **on**, algorithm **HS256**, signing key remains the Supabase JWT secret. Save. Clerk immediately issues HS256 tokens again.
2. **Backend** won't validate those because it's now RS256-only. Revert the `deps.py` change — either `git revert <commit>` or manually restore the HS256 code path from the 2026-04-16 spec. The old `SUPABASE_JWT_SECRET` reference still works because we kept the field on `Settings`.
3. Redeploy the backend. Back to the legacy equilibrium within a release cycle.

Because Supabase's Third-Party Auth provider for Clerk is already configured and inert when tokens arrive HS256-signed, **no Supabase dashboard change is needed to rollback**. Supabase keeps working either way.

## 7. Out of scope

- **Token refresh on the backend.** Clerk tokens are ~60s; the frontend refreshes via `window.Clerk.session.getToken()`. Nothing for the backend to do.
- **Role-based access control.** `role=authenticated` is in every JWT; no admin split exists today. When it does, it gets its own spec.
- **Rate limiting per `sub`.** Belongs at the edge (nginx, Cloudflare, or a FastAPI middleware); not an auth-layer concern.
- **Cookie / session fallback.** Clerk frontends send bearer tokens; cookies are not part of this model.
